Technical Architecture & Engineering Documentation

Um mergulho profundo, vírgula por vírgula, na estrutura que compõe o ecossistema NicheFinder Guru. De requisições API geoespaciais e Web Scraping à renderização de PDF on-the-fly e Automação de Sockets do WhatsApp.

System Architecture

O NicheFinder Guru não é um monolito. É um sistema distribuído projetado para contornar limitações de rede, bloqueios do WhatsApp (Bans) e pesadas cargas de geração de arquivos. A arquitetura divide a Inteligência e a Execução.

// O Fluxo Fundamental da Aplicação

1. [Frontend React] Dispara busca Google Places via Edge Functions.

2. [Web Scraper] Audita o site (ou a ausência dele) e insere no PostgreSQL (Supabase).

3. [Groq AI] Analisa a dor (ex: "Sem site no mapa") e cria a Copy de dor.

4. [PDF Service] Frontend envia payload via HTTP para `localhost:3001` (Node/Puppeteer) que cospe um buffer de PDF.

5. [WA Bot] Processo desacoplado consome a fila do banco e dispara o PDF e o texto via protocolo Baileys (WP Web).

Radar B2B Engine

A fonte primária de dados. Localizado em /src/pages/Radar.tsx, este motor é o responsável pela captação cirúrgica de leads (empresas locais).

Vírgula por vírgula: O Loop de Mapeamento

  • 1. Parâmetros de Entrada: O usuário insere o nicho (ex: "Clínica Odontológica") e a localidade. O sistema permite múltiplas cidades separadas por vírgula.
  • 2. Request Proxy: Para não vazar a chave do Google Maps no front-end, o disparo ocorre através de uma Edge Function no Supabase ou endpoint Node próprio.
  • 3. Coleta de Metadados: A API retorna Nome, Nota, Número de Avaliações, Endereço e o mais crítico: Website.
  • 4. A Lógica "Goldmine" (Sem Site): O radar instantaneamente flagra leads cujo nó `website` retorne nulo. Este é o ativo mais valioso do sistema, pois indica uma empresa que fatura fisicamente, mas é invisível digitalmente.
  • 5. Blacklist Filter: Antes de renderizar na DOM, os dados passam por um array de regex que exclui empresas como "Agências de Marketing", "Sebrae", "Prefeituras".

Technical Auditor

Para leads que possuem um site, o sistema não desiste. Ele procura a falha. O siteAuditor.ts assume o controle.

// Lógica assíncrona do Auditor
const auditResult = await runDeepScan(url);
if (auditResult.ssl === false) {
  flagLeadAs('VULNERABLE_NO_HTTPS');
}
if (auditResult.pageSpeed < 50) {
  flagLeadAs('CRITICAL_SPEED');
}

O auditor faz requisições diretas ao protocolo da URL para checar Handshake SSL. Em seguida, dispara um payload para a API do Google PageSpeed Insights pedindo a métrica mobile (a mais sensível), retornando o Core Web Vital Score de FCP e LCP.

Thermal Sorting (O Score)

Se nós apenas listássemos os leads em ordem alfabética, seríamos uma lista telefônica. O NicheFinder usa o algoritmo Thermal Sorting para empurrar pro topo da fila quem tem mais probabilidade de fechar uma compra.

A Hierarquia de Dor (Quente = Mais Vermelho)

  1. STATUS: SEM_SITE (Peso: 100) - Alvo principal. Sem presença digital.
  2. STATUS: SEM_SSL (Peso: 80) - Site aponta como "Inseguro" no Chrome. Alvo de pânico.
  3. STATUS: SLOW_MOBILE (Peso: 60) - PageSpeed menor que 50. Perde conversão no Ads.
  4. STATUS: HEALTHY (Peso: 10) - O lead está bem servido. Fica no final da fila.

Groq AI Integrations

Integração direta com o modelo llama-3 da Groq (para velocidade absurda de processamento e latência mínima). O script aiService.ts utiliza Template Literals rígidos num system prompt focado em persuasão fria.

Prompt de Conversão Injetado (Exemplo de "Sem Site")

"Você é um especialista em conversão B2B. O cliente ${lead.name} é do nicho ${lead.niche} em ${lead.city}. Ele FATURA fisicamente com ${lead.rating} estrelas, MAS NÃO POSSUI SITE. Monte uma mensagem de WhatsApp casual (máximo 3 linhas) alertando que os concorrentes da cidade estão roubando a demanda de quem busca no Google e não o encontra."

A resposta não é salva estática; caso a estratégia de venda mude, a IA gera um pitch novo no momento do play. Para contornar alucinações de modelos generativos, usamos parâmetros de `temperature` abaixo de 0.4.

The PDF VIP Generator: O Motor Puppeteer

A maior inovação de engenharia do projeto. PDFs gerados client-side travam a aba no navegador e não convertem CSS complexo bem. Nós resolvemos criando um micro-serviço Node.js isolado na porta 3001.

O Ciclo da Geração:

1. Render to String
O Frontend pega um componente React complexo (SitePDF.tsx), converte para HTML String usando renderToStaticMarkup, embute o CSS Tailwind bruto, e envia um POST payload pro servidor 3001.
2. Headless Chrome
O pdf-engine.js usa o Puppeteer rodando de forma invisível. Ele abre uma aba em branco (`page.goto(data:text/html)`), carrega a DOM enviada, injeta fontes do Google e aguarda o networkidle0.
3. Buffer to Base64
O Puppeteer imprime a DOM no formato de impressão (A4 sem margens). O buffer PDF é gerado na RAM, formatado em base64 e retornado no body da resposta HTTP. Sem gravar arquivos no disco.

The Standalone WhatsApp Bot

O frontend do NicheFinder Guru não envia mensagens para o WhatsApp. Fazer envios em massa via web webhooks causa banimento imediato. A solução é o script isolado wa-bot.js baseado no Baileys (Node WhatsApp Web socket).

A Arquitetura de Fila (Queue)

O bot nunca olha pro frontend. Ele se conecta via @supabase/supabase-js na tabela `whatsapp_queue`. Ele escuta o banco com long polling ou Webhooks. Quando o humano no Painel de Automação aperta "Play" em uma lista de Leads Quentes, eles caem nessa fila com status "PENDING".

Flow de Execução Segura (Anti-Ban):

  • → Bot acorda e puxa Lead 1 da Fila.
  • → Checa se o número existe no WhatsApp (onWhatsApp() query).
  • → Simula "Typing..." por X segundos (onde X = len(mensagem) * 0.1).
  • → Envia o PDF em Base64 como document mimeType.
  • → Envia a Copy de Dor gereda pela Groq.
  • → Gatilho de Segurança: Applica um `setTimeout` randomizado entre 45s a 120s antes de olhar pro Lead 2, simulando comportamento estritamente humano padrão.
  • → Marca na `whatsapp_queue` como "SENT".

User Interface & Flow

A interface foi construída seguindo o padrão Dark Mode Brutalist, visando menos distração estética e máxima concentração em dados numéricos e indicadores de falha dos leads. Navegação puramente Sidebar state-based.

A. A Tela do Radar

O painel de inicialização. Possui as entradas para Nicho e Cidades (gerenciadas via Local State do React). Ao iniciar, abre um feed em overlay que simula a requisição TCP caindo (através de animações progressivas usando CSS animate-in e Tailwind slide-in), exibindo cartões modulares indicando as tags vermelhas "SEM SITE".

B. Painel de Leads (CRM)

Tabela construída com suporte a paginação assíncrona. Aqui o Thermal Sorting entra em cena visualmente. Os leads sem site ganham selos de chamas (indicando urgência alta). Nesta aba, o cliente pode clicar no menu de contexto (cabeçalho de ação do Shadcn UI) e selecionar "Auditar Deep Scan" ou individualmente disparar a API de PDF.

C. Painel de Automação (Missão de Controle)

Um espelho do banco de dados `whatsapp_queue`. Ele puxa dados em realtime do Supabase (utilizando subscriptions WebSocket fornecidos pela lib nativa). Não executa nenhuma rotina Node de disparo; ele apenas envia ordens (comandos UPDATE na base) ordenando o wa-bot.js sobre o que fazer e assistindo à contagem das mensagens "Pendentes" e "Enviadas" alterarem dinamicamente.
